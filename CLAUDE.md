# 프로젝트 규칙 (CLAUDE.md)

이 디렉토리(`프로젝트/`) 내 모든 작업은 본 규칙을 절대 준수하며, 임의 생략을 금지합니다.

## 🚀 핵심 개발 파이프라인 (4단계 필수 순서)

사용자 명령에 관계없이 반드시 아래 순서대로 실행하십시오. 단계를 건너뛰는 것은 절대 불가합니다.

### 1단계: 기획 및 스펙 확정
- **① 레퍼런스·기획서 유무 확인**: 작업 시작 전 항상 레퍼런스 URL, 기획서, 디자인 시안 등 참고 자료가 있는지 먼저 묻습니다.
- **② 자료 분석 또는 초안 작성**:
  - 자료가 있으면 → 분석·해석 후 도메인 용어와 MVP 초안을 정리합니다.
  - 자료가 없으면 → 아이디어 기반으로 MVP 초안을 작성합니다.
- **③ `@grill-me` 실행 (항상 필수)**: 자료 유무와 관계없이 반드시 `@grill-me` 스킬을 실행하여 스펙의 빈틈을 채우고 최종 확정합니다. 코딩은 스펙 확정 전까지 절대 금지합니다.

### 2단계: 디자인 시스템 고정
- **실행**: 코딩 전, 전역 스킬 `@ui-ux-pro-max`를 호출하여 서비스 목적에 맞는 UI 테마와 컬러 팔레트를 2~3개 제안하고 사용자의 최종 컨펌을 받으십시오.
- **제약**: 확정된 배색과 가이드는 이후 생성되는 모든 컴포넌트에 누락 없이 상속해야 합니다.

### 3단계: 수직적 작업 분할
- **실행**: 코딩을 시작하지 말고, 확정된 스펙 기반으로 `@to-issues` 스킬을 실행하여 개발 단위를 '수직적 슬라이스(Vertical Slices)'로 분할하십시오.
- **제약**: 분할된 내용을 `docs/` 내 TODO 리스트 또는 GitHub Issues에 선등록하고, 개발 순서에 대해 사용자의 승인을 받으십시오.

### 3.5단계: 리소스 체크리스트 승인
- 코딩 전 **패키지·외부서비스·권한·배포비용**을 표로 정리해 제출. 사용자 승인 전 코딩 금지.
- 거부된 리소스는 대체 방식으로 재설계.

### 4단계: 서브에이전트 기반 TDD 및 프론트엔드 디자인 구현
- **실행**: 승인된 첫 슬라이스부터 순서대로 개발하되, `@tdd` 규칙과 더불어 **전역 스킬 `@frontend-design`을 상시 필수 호출 및 적용**하십시오.
- **서브에이전트 연동**: 구현 루프는 반드시 `@superpowers:subagent-driven-development` 스킬을 작동시키십시오. 메인 컨텍스트 절약을 위해 독립 서브에이전트를 격리 환경에서 실행하십시오.
- **TDD 루프**: 반드시 `RED ➡️ GREEN ➡️ REFACTOR` 단계를 준수하십시오.
- **디자인 제약**: 정형화된 AI 스타일(무분별한 라운딩 및 보라색 그라데이션)을 엄격히 금지하며, 코드를 생성할 때 반드시 `@frontend-design` 스킬을 통해 2단계 테마 기반의 독창적인 타이포그래피와 인터랙티브 애니메이션을 반영하십시오.
- **루프 안전장치**: 동일한 테스트 실패가 3회 반복되면 자율 수정을 즉시 중단하고, 에러 로그와 함께 사용자에게 가이드를 요청하십시오.

### [필수 후처리] 시작 배치 파일 생성
- 새 프로젝트 생성 시 루트에 `시작 {포트번호}.bat` 파일을 반드시 생성하십시오. (기존 포트와 중복 금지)
- **파일 내용 구조**:
  ```bat
  @echo off
  chcp 65001
  cd /d "%~dp0"
  start http://localhost:{포트번호}
  {서버 시작 명령어 (예: npm run dev)}
  ```

## 📂 폴더 구조 및 운영 원칙

### 1. 표준 디렉토리 트리
`src/`(소스), `tests/`(TDD 필수), `docs/`(설계/이슈), `public/`(정적 에셋). 단, 프레임워크(Next.js, Vite 등) 표준 구조가 있을 경우 이를 최우선 준수합니다. 하위 프로젝트 루트에는 본 문서를 상속한 `CLAUDE.md`를 필수 생성하십시오.

### 2. 디자인 작업 제약
아이콘 생성·수정 및 UI에 아이콘 요소가 포함된 모든 작업에서 `@icon-design` 스킬을 적용하십시오.

### 3. 작업 위치 및 목적별 분류
모든 작업은 `프로젝트/` 하위에서만 진행하며, 시작 전 목적을 확인하여 아래 분류 폴더 내에 프로젝트를 생성하십시오. 목적 변경 시 폴더 이동 후 사용자에게 보고하십시오.

- `for_Local/`: 배포가 없는 로컬 전용 작업
- `for_Release/`: 실제 배포 및 운영용 프로젝트
- `for_Drop/`: MVP 검증용 일회성 프로젝트
